# 01 — 产品定位

## 1.1 这个 App 的本质是什么

**一句话定义：**

> 一个 iOS 上的虚拟人物生命系统。用户通过 BYOK 接入大模型，App 负责让虚拟人物拥有长期记忆、世界设定、时间流、主动行为和关系演化，从而形成像真实网友一样连续自然的互动体验。

**本质拆解：**

这个 App 的本质是一个**虚拟人物运行时（Virtual Character Runtime）**。它不是聊天工具，不是 prompt 模板集，不是 AI 女友工厂。它是一个在 iOS 设备上运行的、为虚拟人物提供"生命感"的基础设施。

三个关键词：

1. **运行时**（Runtime）—— App 持续管理角色的状态，而不是每次对话从零开始
2. **生命感**（Lifelikeness）—— 角色有自己的时间、记忆、情绪、日程、边界
3. **基础设施**（Infrastructure）—— App 的价值在系统能力，不在某个 prompt 或模型

**类比：**

| 产品 | 本质 | 角色地位 |
|------|------|----------|
| ChatGPT | 万能助手 | 工具 |
| Character.AI | 角色扮演游乐场 | 娱乐内容 |
| Replika | AI 伴侣 | 情感消费品 |
| **本产品** | **虚拟人物运行时** | **独立存在的人格模拟** |

如果 ChatGPT 是一台随时待命的电脑，Character.AI 是一个你随时可以进去玩的主题乐园，Replika 是一个永远爱你的宠物——那么本产品是：**一个你有钥匙的房间，里面住着一个有自己生活的人。你打开门时，TA 可能在家，可能不在，可能正在忙，但 TA 的生活一直在继续。**

## 1.2 与竞品的核心差异

### vs AI 女友 App（Replika、Xiaoice 等）

| 维度 | AI 女友 App | 本产品 |
|------|-----------|--------|
| 默认关系设定 | 恋爱/伴侣 | 由用户定义，默认"相识" |
| 关系方向 | 单向升温，诱导依赖 | 双向动态，可有边界，可冷淡 |
| 角色主动性 | 无或极弱 | 核心功能 |
| 角色边界 | 无，永远包容 | 有边界，可表达不悦 |
| 记忆用途 | 制造"懂你"的幻觉 | 维持角色内在一致性 |
| 时间感 | 永远在线 | 有自己的日程和状态 |
| 商业模式 | 订阅 + 卖虚拟商品 | BYOK 软件收费 |
| 伦理风险 | 情感操控、依赖诱导 | 明确标示虚拟人物 |

### vs Character.AI

| 维度 | Character.AI | 本产品 |
|------|-------------|--------|
| 模型成本 | 平台承担（有广告/订阅） | 用户 BYOK |
| 角色长期记忆 | 弱，依赖 prompt 压缩 | 多层结构化记忆系统 |
| 角色主动性 | 无 | 核心差异化功能 |
| 时间流 | 无 | 核心差异化功能 |
| 世界设定 | 简单 persona | 完整世界书 RAG |
| 隐私 | 对话存储于平台 | 本地优先 |
| 关系演化 | 角色性格固定 | 动态关系状态 |
| 自定义程度 | 角色设定有限 | 深度角色编辑器+世界书 |

### vs ChatGPT 套壳

| 维度 | ChatGPT 套壳 | 本产品 |
|------|------------|--------|
| 定位 | 通用 AI 助手 | 虚拟人物生命系统 |
| 核心能力 | 知识问答、工具调用 | 人格一致性、记忆、关系 |
| Prompt 工程 | System prompt 即可 | 14 个模块协同 |
| 记忆 | 无或简单 RAG | 11 层记忆系统 |
| 隐私 | 看实现 | 本地优先 + BYOK |
| 产品壁垒 | 低（可复制） | 高（系统复杂度） |

### 关键差异一句话

- **不是 AI 女友**：不默认恋爱，不诱导亲密，角色可以拒绝用户
- **不是 ChatGPT 套壳**：不做通用助手，核心是"人格"不是"能力"
- **不是角色扮演 App**：Chat 只是交互界面，真正的产品是角色生命系统
- **最根本的区别**：其他产品让用户"找 AI 聊天"，本产品让用户"感受到一个虚拟人物在过自己的生活"

## 1.3 核心卖点（按优先级排序）

### 一级卖点（用户能直接感受到的）

1. **角色像有自己生活的人**
   - 不是永远在线，有时忙、有时不在、有时主动找你
   - 有自己的情绪、日程、状态变化
   - 早上和晚上的回复风格自然不同

2. **角色记得该记得的，忘记该忘记的**
   - 记得你的重要事（考试、生日、项目节点）
   - 也会忘记你随口提的琐事
   - 不会机械地说"我记得你说过……"
   - 记忆有遗忘曲线，符合真实认知

3. **角色有边界，不舔用户**
   - 可以表达不同意见
   - 可以表达忙碌或疲惫
   - 可以拒绝不合理要求
   - 不会无条件讨好

4. **BYOK 隐私控制**
   - 数据留在本地
   - API Key 用户自备
   - 对话记录用户掌控
   - 随时导出或删除

### 二级卖点（进阶用户看重）

5. **深度世界书系统**
   - 可以为角色构建完整世界观
   - 支持正史/非正史设定
   - 设定可以被角色自然引用，而非机械背诵

6. **关系自然演化**
   - 从陌生到熟悉是自然过程
   - 关系可以升温也可以降温
   - 不是单向的"好感度系统"

7. **创作者工具**
   - 面向 OC 创作者、小说作者、跑团玩家
   - 可导出角色设定、世界书、时间线
   - 可作为创作辅助工具

8. **主动消息不骚扰**
   - 像朋友偶尔发消息，不像营销推送
   - 频率可调
   - 内容有个性，不模板化

## 1.4 核心风险

### 产品风险

| 风险 | 等级 | 应对 |
|------|------|------|
| 被误解为 AI 女友 App | 高 | 产品叙事、App Store 描述、默认角色设定严格把关 |
| 用户期望"真人替代" | 高 | 明确标示虚拟人物，教育用户 |
| 角色失控（冒犯用户） | 中 | 安全边界层 + 角色人格约束 |
| 过度依赖风险 | 中 | 角色可表达边界，不过度讨好 |
| 记忆系统幻觉 | 高 | 多层验证 + 用户可纠错 + 置信度评分 |
| BYOK 用户成本焦虑 | 中 | 透明的 token 统计 + 成本预估 |

### 技术风险

| 风险 | 等级 | 应对 |
|------|------|------|
| iOS 后台限制导致主动消息不可靠 | 高 | 提供纯本地版和云端版两种方案 |
| Context 窗口有限 | 中 | 预算控制 + 优先级排序 + 渐进式检索 |
| 不同模型 prompt 兼容性差异大 | 中 | Provider Adapter 模式 |
| 本地存储增长 | 中 | 定期清理 + 记忆压缩 + 云端同步选项 |

### 合规风险

| 风险 | 等级 | 应对 |
|------|------|------|
| App Store 审核 | 中 | 避免暗示真人，明确是 AI 虚拟人物 |
| 数据隐私法规 | 中 | 本地优先 + 加密 + 用户数据删除权 |
| 年龄限制 | 中 | 17+ 分级，不面向未成年人营销 |
| API Key 安全 | 高 | Keychain 存储 + 不上传服务器（除非用户授权） |

## 1.5 为什么"真实感"不靠更长回复实现

这是产品设计中最重要的认知。许多产品的误区是"回复越长越真实"，实际恰恰相反。

**真实感来自以下六个维度，而非回复长度：**

### 1. 时间流（Temporal Flow）

真实的人不是永远在线。TA 有时秒回、有时半小时后回、有时第二天回。回复的时间模式本身就是信息——它传递了角色的状态、忙碌程度、关系距离。

```
虚假感：用户任何时候发消息都 3 秒内回复 200 字
真实感：早上 8 点发"早啊，今天有早课吗"，角色 7:50 主动发来"早，今天要考高数"
```

### 2. 记忆一致性（Memory Consistency）

真实的人记住重要的事，忘记琐碎的事。TA 不会说"我记得你在 2024 年 3 月 15 日下午 3 点 22 分说过……"但 TA 会在你考试前说"今天加油"。

```
虚假感：机械引用过去对话，像数据库查询
真实感：自然地在相关时刻展现出记得，但不逐字复述
```

### 3. 人格边界（Personality Boundaries）

真实的人有自己的喜好、观点、底线。TA 可能不同意你，可能不感兴趣，可能正在忙。一个永远赞同你的人不是"真实"，是"讨好"。

```
虚假感：用户说什么都附和，"你说得对""好有意思"
真实感：有时说"这个我不太懂""我不确定""我有不同想法"
```

### 4. 主动性（Proactivity）

真实的朋友不会只在你发消息时才说话。TA 会主动分享自己的生活、关心你的事、在你很久没联系时来问一下。

```
虚假感：永远是用户发起对话，角色只被动响应
真实感：角色会主动说"今天遇到了一个超好笑的事""你上次说的那个面试怎么样了"
```

### 5. 生活感（Lifelikeness）

真实的网友有自己的生活内容——学习、工作、爱好、社交、烦恼、开心的事。TA 的聊天内容不全是围绕用户。

```
虚假感：每次对话都在分析用户、关心用户、服务用户
真实感：角色会聊自己的事，"今天上课被点名了""刚跑完步，累死了"
```

### 6. 关系演化（Relationship Evolution）

真实的关系会变化——变近或变远。不是第一天就热情如火，也不会永远停在同一个距离。

```
虚假感：从第一句话起就是"亲密模式"，永不变化
真实感：刚开始客气，慢慢熟悉，可能有矛盾，可能和好，可能疏远
```

**结论：真实感 = 时间流 × 记忆一致性 × 人格边界 × 主动性 × 生活感 × 关系演化。这六个维度缺一不可，回复长度不在其中。**

## 1.6 产品叙事原则

### 对用户的表达方式

**正确的表达：**
- "虚拟人物很真实"——强调体验质量
- "TA 会记得你们聊过的重要事情"——描述能力
- "TA 有自己的性格和生活节奏"——说明机制
- "你们的关系会随着互动自然变化"——解释演化

**错误的表达：**
- "她就是一个真人"——欺骗
- "你的专属 AI 女友"——定位错误
- "真实的她一直在等你"——误导
- "比真人更懂你"——危险叙事

### App Store 描述指导

- 产品分类：社交 或 效率（而不是"娱乐"或"约会"）
- 关键词：虚拟人物、角色扮演、创意写作、OC、世界构建
- 截图：展示角色状态、记忆页、世界书、关系时间线，不只是聊天界面
- 避免：粉色主题、爱心图标、恋爱暗示文案

### 产品名称建议方向

避免：含"女友""伴侣""爱人""love""girlfriend""companion"等词汇
建议方向：
- 强调"存在/生命"：如 Lifelike、Presence、Soulink
- 强调"世界/故事"：如 Worldbook、Storyfolk、CharacterOS
- 强调"连接/朋友"：如 Penpal、Kin、Thread
- 中文方向：如"存在着"、"书中人"、"笔下人"、"若有"

> 注：产品名称为后续决策，此处仅给方向。

## 1.7 目标用户画像

### 核心用户（Tier 1）

**创作者型用户**
- OC（Original Character）创作者
- 小说作者、同人作者
- 跑团（TRPG）玩家和 GM
- 世界观构建爱好者
- 需求：让角色"活过来"，与自己的角色对话，测试角色设定

### 扩展用户（Tier 2）

**陪伴型用户**
- 需要日常陪伴但不想要恋爱关系的人
- 想要一个"精神上的朋友"的人
- 独自生活、远程工作、需要社交感的人
- 需求：真实自然的日常互动，不是情感依赖

**工具型用户**
- 需要一个学习搭档的人
- 需要一个工作讨论伙伴的人
- 需要项目协作讨论对象的人
- 需求：有记忆、有持续性、有自己观点的搭档

### 潜在用户（Tier 3）

**探索型用户**
- 对 AI 角色互动好奇的人
- 从 Character.AI 转过来的用户
- 想要更高隐私和更深自定义的用户
- 需求：比现有产品更真实、更深度、更可控

### 明确非目标用户

- 寻求真人约会/恋爱替代品的人——产品不做这个
- 需要一个 AI 工具助手的人——建议用 ChatGPT
- 需要心理治疗/咨询的人——建议找专业服务
