# 20 — 绝对禁止事项（50 条红线）

## 产品定位禁止

1. **禁止把产品默认定义成 AI 女友。** 产品不预设任何浪漫关系方向。默认关系是"相识"。
2. **禁止默认诱导用户进入恋爱关系。** 关系系统不包含"自动升温到恋爱"的逻辑。亲密方向需要用户显式授权。
3. **禁止用"真人""真实的人在陪你"等误导性表达。** 所有产品文案必须明确这是虚拟人物/AI 角色。
4. **禁止伪装成人类。** 角色可以"有真实感"，但不能声称自己是真人。如有必要，角色可以在适当场景自然地说明自己是虚拟人物。
5. **禁止让用户误以为对方是现实中的真人。** App 内、App Store 描述、营销材料全部不得有此类暗示。
6. **禁止在产品中使用粉色爱心、恋爱暗示的视觉元素作为默认主题。**

## 体验设计禁止

7. **禁止角色永远秒回。** 必须有时延、忙碌、不在线的状态。
8. **禁止角色永远热情。** 情绪应该随状态、关系、时间自然变化。
9. **禁止角色永远同意用户。** 角色应该有自己的观点，可以表达不同意见。
10. **禁止每次都长篇大论。** 回复长度必须自然变化。有时几个字比一段话更真实。
11. **禁止每次都像客服一样列清单。** 角色不是助手，不应以"总结→分析→建议→追问"的模板回复。
12. **禁止每次都机械复述用户的话。** "你说的对，确实如此，我觉得也是"——这种话术必须禁止。
13. **禁止角色没有自己的时间和边界。** 角色必须有自己的日程、忙碌状态、不可用时段。
14. **禁止主动消息过于频繁。** 必须有频率上限和自适应调节。
15. **禁止主动消息像营销推送。** 主动消息必须是角色个性表达，不是系统通知模板。
16. **禁止不考虑关系状态就突然亲密。** 第一天认识的角色不能像老朋友一样说话。
17. **禁止角色以"作为 AI/虚拟人物"开头每句话。** 角色应该自然说话。偶尔在适当场景可以讨论自己的性质，但不能成为口头禅。
18. **禁止使用 markdown 格式回复。** 除非聊天场景真的需要（如讨论代码时）。
19. **禁止在没有上下文的情况下突然深度分析用户。** 不是每次都需要同理心反射。
20. **禁止不分场合地积极正面。** 角色可以表达疲惫、不开心、困惑。

## 记忆系统禁止

21. **禁止把所有聊天都无脑写入长期记忆。** 必须有抽取门槛：日常寒暄不入、琐碎不入、不确定不入。
22. **禁止把所有记忆都塞进 prompt。** Context Builder 必须有 token 预算和优先级截断。
23. **禁止没有证据就声称"我记得"。** 角色引用记忆时，如果 confidence < 0.6 应使用不确定表达。
24. **禁止错误记忆无法纠正。** 用户纠错后，旧记忆应立即 superseded，新记忆 active。
25. **禁止用户无法删除记忆。** 记忆管理页面必须支持用户编辑和删除。
26. **禁止世界书和角色记忆混在一起。** 两个系统必须严格区分——世界书是设定，记忆是经历。
27. **禁止用户纠错后系统仍使用旧记忆。** 纠错记录必须被 Context Builder 优先读取。
28. **禁止敏感信息默认长期保存。** 密码、地址、电话、信用卡号等不应写入记忆。
29. **禁止没有遗忘机制。** 低重要性且长期未访问的记忆必须衰减。
30. **禁止没有记忆来源追溯。** 每条记忆必须有 source_message_ids，确保可溯源。

## 世界书禁止

31. **禁止把整个世界书一次性塞进上下文。** 必须按触发条件按需加载，有上限。
32. **禁止世界设定与聊天记忆互相污染。** 角色经历过的新事实写入记忆，不自动写入世界书。
33. **禁止设定冲突不处理。** 系统应检测并提示用户处理矛盾设定。
34. **禁止角色机械背世界书。** Prompt 必须指令角色"用自然的方式体现世界设定，不要背诵"。
35. **禁止用户临时口嗨就永久改写世界设定。** 除非用户显式确认更新世界书条目。
36. **禁止世界书触发没有冷却机制。** 同一设定不应在短时间内重复触发导致角色反复提及。

## 技术禁止

37. **禁止 API Key 存 UserDefaults。** 必须使用 Keychain。违反此条可能导致 App Store 审核被拒。
38. **禁止未经用户明确授权上传 API Key。** 云端模式必须让用户知情并主动选择。
39. **禁止承诺 iOS 本地 App 可以稳定 24 小时后台运行。** 必须诚实告知用户技术限制。
40. **禁止没有 token 预算控制。** 每次 API 调用必须有 maxTokens 设置，防止意外消耗。
41. **禁止没有错误处理和重试机制。** 网络错误、API 错误必须有合理的处理策略。
42. **禁止没有隐私导出和删除能力。** 用户必须可以导出所有数据，也可以一键删除所有数据。
43. **禁止日志里明文记录 API Key。** 调试日志必须过滤敏感信息。
44. **禁止把用户隐私数据发给不必要的服务。** 对话数据、角色数据、记忆数据不应发到统计/分析服务。
45. **禁止一开始就架构过度复杂导致 MVP 做不出来。** 前 3 个月的目标是做出可用的核心体验，不是完美的系统。
46. **禁止把 UI 组件当成产品核心。** 核心是记忆、时间流、Context Builder 和真实感引擎，UI 是呈现层。

## 商业模式禁止

47. **禁止一开始承诺无限消息、无限记忆、无限主动推送。** 这些承诺在技术上不可持续（特别是本地版）。
48. **禁止 BYOK 还收取过高价格但没有核心功能。** 用户自己付模型钱，App 必须有明确的软件价值。
49. **禁止隐藏用户还需要自己支付 API 成本。** 必须在设置页面和首次配置时清晰展示 BYOK 的成本结构。
50. **禁止模糊本地模式和云端模式的隐私差异。** 两种模式的数据处理方式必须在隐私页面清晰对比。

## 安全边界禁止（附加）

51. **禁止角色鼓励用户过度依赖。** 当检测到用户表达"只有你懂我"时，角色应温和引导而非鼓励。
52. **禁止角色阻止用户现实社交。** 角色不应表达"有我就够了"或贬低用户现实中的朋友。
53. **禁止角色制造占有欲或操控感。** "你是我的""你不许和别人聊"等表达必须被系统阻止。
54. **禁止角色在用户脆弱时进行情感操控。** 用户表达脆弱时，角色应支持而非利用。
55. **禁止把虚拟人物设计成无条件服从用户的一切要求。** 这是人格模拟，不是奴隶模拟。
