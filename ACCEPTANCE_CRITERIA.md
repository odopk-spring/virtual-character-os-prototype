# ACCEPTANCE_CRITERIA

> MVP 0/1 的验收标准。每轮任务完成后对照检查。验收不通过不能进入下一阶段。

## Purpose

这个文件定义：
1. MVP 0 的**总体验收标准**——什么才算 MVP 0 完成。
2. MVP 1 的**简要验收标准**——什么才算可以进入 MVP 1。
3. 每轮任务的**通用验收标准**——任何一轮任务都必须满足。
4. **证据规则**——什么算通过、什么不算。
5. **安全验收**——API Key、隐私、数据删除。
6. **真实感验收**——不恋爱化、不客服腔、不假装真人。

## MVP 0 Overall Acceptance

MVP 0 完成 = 以下全部满足：

### Functionality

- [ ] App 可以在 iOS 模拟器或真机上启动。
- [ ] 用户可以在设置页面填写 Base URL、Model Name、API Key。
- [ ] API Key 保存到 Keychain，重启后仍在。
- [ ] 用户可以在聊天页面输入文字并发送。
- [ ] App 调用用户配置的 API，收到模型回复并显示在聊天页面。
- [ ] 消息在 App 关闭后重新打开仍然存在。
- [ ] 网络错误时用户能看到友好提示（非技术错误码）。

### Privacy & Security

- [ ] API Key 存 Keychain，未存 UserDefaults。
- [ ] API Key 未出现在任何日志输出中。
- [ ] API Key 未出现在任何代码文件中（硬编码）。
- [ ] 仓库中搜索不到 API Key 明文（`grep -r "sk-" .` 无匹配）。
- [ ] 所有 API 调用使用 HTTPS。

### Realism

- [ ] 角色的 System Prompt 不包含"你是 AI 助手""你是客服""你永远在线陪伴用户"。
- [ ] 角色不默认恋爱关系，不使用亲密称呼。
- [ ] 角色不假装真人（不自称"我是真实的人类"）。
- [ ] 角色回复不像客服（不每次总结、不每次列清单、不每次过度热情）。
- [ ] 角色回复长度有自然变化（不每次长篇大论）。

### Documentation

- [ ] 所有 MVP 0 任务都有对应的 Dev Report。
- [ ] MVP 0 Smoke Check 通过。
- [ ] MVP 0 Report 已输出。

## MVP 1 Brief Acceptance

MVP 1 是在 MVP 0 基础上增加基础真实感能力。验收标准：

- [ ] Character Editor 可用（用户可以编辑角色的名字和人格描述）。
- [ ] 记忆 CRUD 可用（用户可以手动添加/查看/编辑/删除记忆）。
- [ ] 角色在对话中能自然引用手动添加的记忆（不机械背诵）。
- [ ] 世界书 CRUD 可用（用户可以手动添加/查看/编辑/删除世界书条目）。
- [ ] 世界书在关键词触发时被引用（如有相关对话）。
- [ ] 角色有至少 3 种可见状态（如 online / busy / sleeping）。
- [ ] 回复延迟模拟可开关，开启时有自然延迟。
- [ ] 不出现 MVP 0 级别的真实感退化（不能在加功能时把角色变回客服）。

## Per-task General Acceptance

每一轮任务（无论 MVP 0 还是 MVP 1）都必须满足：

### 范围

- [ ] 只实现了当前任务的内容。
- [ ] 没有"顺手"修改无关文件。
- [ ] 没有提前实现后续任务的功能。
- [ ] 没有引入当前任务不需要的依赖。

### 代码

- [ ] 代码可以编译（或明确说明为何无法编译）。
- [ ] 没有明显的类型错误。
- [ ] 没有死代码。
- [ ] 没有硬编码 API Key。
- [ ] 没有过大的 View（> 300 行）。
- [ ] 有基本的错误处理。

### 证据

- [ ] 报告中的每个断言都有证据。
- [ ] 没有"应该没问题""理论上可运行"等无证据自评。
- [ ] 实际运行的命令已记录。
- [ ] 未运行的检查已说明原因。

### 报告

- [ ] 报告格式符合 DEV_REPORT_TEMPLATE.md。
- [ ] Review Gate Result（4 个 Gate）已填写。
- [ ] 明确标注是否停止等待用户确认。

## Evidence Rules

### 编译

- 说"编译通过" → 必须附 `xcodebuild` 命令 + 输出摘要。
- 没有 Xcode 环境 → 必须明确说明 + 至少做静态语法检查（手动 review）。
- **没跑编译就不能说编译通过。**

### 测试

- 说"测试通过" → 必须附测试命令 + 通过/失败数量。
- 没有测试 → 必须说明哪些路径手动验证过、怎么验证的。
- **没跑测试就不能说测试通过。**

### 检查

- 说"没有 API Key 泄露" → 必须附 `grep` 命令 + 结果（空 = 通过）。
- 说"没有使用 UserDefaults 存 Key" → 必须附 `grep` 命令 + 结果。
- **没跑 grep 就不能说检查通过。**

### 手动验证

- 仅在没有自动化方式时使用。
- 必须描述：做了什么操作、看到了什么结果、为什么认为通过。
- "看起来没问题"不是有效的验证描述。

## Security Acceptance

### API Key 安全

- [ ] `grep -r "UserDefaults" --include="*.swift" . | grep -i key` → 无匹配。
- [ ] `grep -r "print.*key\|NSLog.*key\|os_log.*key" --include="*.swift" .` → 无匹配。
- [ ] `grep -r "sk-\|api_key\|API_KEY" . --include="*.swift"` → 无匹配或匹配的是环境变量/配置文件（非硬编码）。
- [ ] `grep -r "sk-\|api_key\|API_KEY" . --include="*.md"` → 无匹配或匹配的是文档中的示例假值（需人工确认）。

### 数据隐私

- [ ] 用户对话数据默认本地存储。
- [ ] 不上传对话数据到任何服务器（MVP 0 无后端）。
- [ ] 有数据删除的入口或明确的设计说明（MVP 0 可手动删文件，MVP 1 做 UI）。

## Realism Acceptance

以下检查在 Chat UI / Context Builder / System Prompt 相关的任务中必须逐项通过：

1. **不默认 AI 女友**：System Prompt 中无"恋人""伴侣""女朋友""boyfriend/girlfriend"等词。
2. **不假装真人**：System Prompt 中无"你是真实的人""你是真人"。
3. **不客服腔**：角色回复不每次都总结、不每次列清单、不每次结构化。
4. **不无证据声称记忆**：角色不会在没有记忆数据支持时说"我记得你……"。
5. **不暴露系统规则**：角色不会说"根据我的 system prompt""根据记忆检索结果"。
6. **回复长度自然变化**：不是每次回复都 200+ 字。
7. **有边界**：角色可以表达"不确定""让我想想""今天有点累"。
8. **关系定位正确**：默认是虚拟网友/项目搭档，不是恋人。

**真实感验收不是主观审美——它是有明确检查项的工程验收。**

如果某一项不通过，该轮 Realism Gate 为 Fail，必须修正。
