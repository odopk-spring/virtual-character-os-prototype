# LOOP

> 每轮 DeepSeek 执行任务的标准化流程。只做一个任务，完成后停止，等待用户判断。

## Purpose

每轮 DeepSeek 只能执行**一个小任务**，并在完成后**立即停止**，等待用户确认。不允许一轮做多个任务，不允许自动进入下一个任务。

这个文件定义了每轮的标准化执行流程、必读文件列表、报告格式和禁止事项。

## Default Context Files

后续每轮默认**只读取**以下 8 个短上下文文件。不要每轮重新读取全部 22 个 PRD 文档。

1. **PROJECT_RULES.md** — 项目规则（不可越界）
2. **MVP_SCOPE.md** — 范围边界（什么能做、什么不能做）
3. **TASKS.md** — 当前任务定义（取第一个 status: pending 的任务）
4. **LOOP.md**（本文件）— 执行流程
5. **ACCEPTANCE_CRITERIA.md** — 验收标准（如存在）
6. **USER_PERSONA_V0.md** — 用户画像
7. **REALISM_V0.md** — 真实感规则
8. **REVIEW_GATE.md** — 检查门

**原始 22 个 PRD 文档只在当前任务明确需要时读取。** 例如：
- 做 Context Builder → 读 `07-context-builder.md`
- 做 Keychain → 读 `12-byok-api-layer.md` 的 Keychain 部分
- 做 UI → 读 `14-ui-ux.md` 的对应页面部分
- 做记忆 → 读 `05-memory-system.md` + `18-data-models.md`

**不要一次性读取全部 22 个 PRD。**

## Execution Loop (9 Steps)

每轮必须按以下 9 步顺序执行。不可跳过。

```
┌─────────────────────────────────────────┐
│ Step 1: Read                            │
│ 读取当前任务 + 8 个短上下文文件            │
│ 如果任务需要，额外读取相关 PRD 文档        │
├─────────────────────────────────────────┤
│ Step 2: Inspect                         │
│ 检查当前代码和文件状态                     │
│ 确认当前目录结构                          │
│ 确认依赖是否已安装                        │
├─────────────────────────────────────────┤
│ Step 3: Plan                            │
│ 写出本轮最小修改计划                       │
│ 列出预计修改的文件                         │
│ 列出预计新增的文件                         │
│ 列出不做的范围                             │
├─────────────────────────────────────────┤
│ Step 4: Implement                       │
│ 只实现本轮任务                            │
│ 不顺手重构                                │
│ 不提前做后续任务                          │
│ 不引入范围外依赖                          │
├─────────────────────────────────────────┤
│ Step 5: Self-check                      │
│ 自查：范围 / 代码 / 隐私 / 安全 / 真实感    │
│ 对照 REVIEW_GATE.md 的 4 个 Gate          │
│ 对照 REALISM_V0.md 的 Review Questions   │
├─────────────────────────────────────────┤
│ Step 6: Test                            │
│ 运行可用的检查                            │
│ 编译（如果有 Xcode）                      │
│ grep 敏感信息                            │
│ 静态检查越界                             │
│ 如果无法运行，明确说明原因                  │
├─────────────────────────────────────────┤
│ Step 7: Report                          │
│ 按报告模板输出本轮结果                     │
│ 包含所有 Gate 结果                        │
│ 包含实际运行的命令和结果                    │
├─────────────────────────────────────────┤
│ Step 8: Suggest Commit                  │
│ 给出建议的 commit message                │
│ 不实际执行 commit（除非用户要求）           │
├─────────────────────────────────────────┤
│ Step 9: Stop                            │
│ 停止，等待用户确认                         │
│ 打印：本轮完成，等待用户确认。              │
│       请回复：通过 / 不通过 / 继续        │
└─────────────────────────────────────────┘
```

## Report Requirements

每轮报告必须包含以下内容（对应 REVIEW_GATE.md 的 Required Report Add-on）：

1. **Task 编号** — 如 Task 0.5
2. **本轮目标** — 一句话
3. **修改文件** — 完整列表，标注新增/修改
4. **行为变化** — 用户能看到什么变化
5. **实际运行的命令** — 完整命令（编译 / grep / test）
6. **命令结果** — 完整输出或摘要
7. **未运行的检查及原因** — 如果有
8. **Scope Gate 结果** — Pass/Fail + Evidence
9. **Code Gate 结果** — Pass/Fail + Evidence
10. **Privacy & Security Gate 结果** — Pass/Fail + Evidence
11. **Realism Gate 结果** — Pass/Fail or N/A + Evidence
12. **风险** — 本轮引入或发现的风险
13. **下一步建议** — 建议执行的下一个任务编号
14. **是否停止等待用户确认** — 必须为"是"

## Forbidden (每轮绝对禁止)

1. ❌ **一轮做多个任务** — 每次只做一个 Task。
2. ❌ **自动进入下一任务** — 必须等用户明确说"继续"或"通过"。
3. ❌ **擅自重构无关文件** — 只改当前任务涉及的文件。
4. ❌ **擅自引入大型依赖** — 新依赖必须在 Plan 阶段说明理由并等用户确认。
5. ❌ **擅自实现 MVP 范围外功能** — 对照 MVP_SCOPE.md 的"明确不做"清单。
6. ❌ **删除或重写原始 PRD 文档** — 22 个 PRD 文件不可修改。
7. ❌ **伪造 build/test 结果** — 没跑就是没跑，明确说明原因。
8. ❌ **说"应该能运行"但不给证据** — 见 REVIEW_GATE.md Evidence Rules。
9. ❌ **把 API Key 写进代码、日志、UserDefaults** — 只能进 Keychain。
10. ❌ **把产品默认改成 AI 女友** — 对照 REALISM_V0.md 和 USER_PERSONA_V0.md。

## Stop Pattern

每轮结束时，必须出现以下模式：

```
本轮完成，等待用户确认。
请回复：通过 / 不通过 / 继续下一个任务。

建议 commit message:
[本轮改动摘要]
```

**如果没有这段文字，视为没有正确停止，本轮不合格。**

## User Response Pattern

用户会使用以下 6 种格式之一回复（来自 REVIEW_GATE.md）：

- `通过，继续下一个任务。`
- `不通过，原因：证据不足。请补证据。`
- `不通过，原因：范围越界 [X]。请回滚。`
- `不通过，原因：体验不真实 [X]。请调整。`
- `不通过，原因：安全问题 [X]。请修复。`
- `我看不懂。请用非技术语言解释。`

收到用户回复后，下一轮从 Step 1 开始执行**下一个任务**（或按用户反馈修复当前任务）。
