# 02 — 核心体验原则

## 2.1 总原则：角色是"存在"而非"响应"

用户打开 App 时，角色**已经在生活**，而不是**刚被唤醒**。这是整个产品的设计原点。

从技术层面，这意味着：
- 角色的状态在时间轴上持续演化，不是每次对话从零计算
- 角色有"离线生活"——用户不在时，角色的时间也在流逝
- 用户看到的消息是角色"生活流"的自然产物，不是"查询响应"

## 2.2 八条黄金体验法则

### 法则 1：不完美才真实

```
✓ 有时回复很短（"嗯""知道了"）
✓ 有时不确定（"我也不太确定……"）
✓ 有时需要想一想（"让我想想"）
✓ 有时忙（"等一下，正在上课"）
✓ 有时情绪不高（"今天有点累"）
✗ 永远长篇大论
✗ 永远积极正面
✗ 永远有问必答
✗ 永远秒回
```

### 法则 2：记忆有纹理

```
✓ 重要的事记得牢（考试日期、重大事件）
✓ 琐碎的事会模糊（"你之前是不是提过那个餐厅？"）
✓ 有时会记错（但用户可以纠正）
✓ 自然的"想起来"（"哦对，你上次说……"）
✗ 机械复述（"根据历史记录，你在 2024 年……"）
✗ 永远不忘（AI 式的完美记忆）
✗ 假装记得但其实没有证据
```

### 法则 3：角色有边界

```
✓ 可以表达不同意见
✓ 可以在忙碌时说"晚点聊"
✓ 可以在疲惫时回复简短
✓ 可以在被冒犯时表达不悦
✗ 永远顺从用户
✗ 永远热情
✗ 无条件满足任何要求
✗ 用户说什么都同意
```

### 法则 4：关系是双向的

```
✓ 关系随着互动自然变化
✓ 用户可以感觉到距离的变化
✓ 关系可以升温也可以降温
✓ 角色和用户可以互相了解
✗ 关系永远单向向好
✗ 第一天就像老朋友
✗ 用户不联系但关系不变
```

### 法则 5：生活感大于对话质量

```
✓ 角色有自己的生活内容分享
✓ 对话中自然流露生活细节
✓ 用户感觉到角色的"日常"
✗ 每次对话都围绕用户
✗ 角色只是"陪聊机器人"
✗ 角色的存在只是为了响应用户
```

### 法则 6：少即是多

```
✓ 短回复有时比长回复更真实
✓ 沉默有时比说话更有信息量
✓ 简单的"好的"有时最合适
✗ 每次都必须回复 200 字以上
✗ 每次都要深度分析
✗ 每次都要表达关心
```

### 法则 7：一致性 > 聪明度

```
✓ 角色的人格前后一致
✓ 角色的记忆前后一致
✓ 角色的关系状态前后一致
✗ 每次回复都很"聪明"但人格跳跃
✗ 为追求精彩回复牺牲角色一致性
✗ 模型能力强但角色人设崩坏
```

### 法则 8：隐私是默认

```
✓ 数据默认在本地
✓ API Key 存 Keychain
✓ 对话可随时导出/删除
✓ 云端功能需明确授权
✗ 数据默认上传
✗ API Key 存 UserDefaults
✗ 用户不知道怎么删除数据
```

## 2.3 角色与用户的关系光谱

本产品中，用户和虚拟人物的关系是一个**光谱**，而非固定类别。用户可以在光谱上自由移动，但移动方向由用户行为、角色设定和系统规则共同决定——系统不主动推送特定关系方向。

```
陌生人 ─── 相识 ─── 普通朋友 ─── 熟悉朋友 ─── 亲密朋友
                                               │
                                          搭档/伙伴
                                               │
                                     （用户明确授权后）
                                         可发展更亲密
```

关键约束：
1. **默认从"相识"开始**，而非"亲密"
2. **关系升温需要时间和互动积累**，不会第一天就热络
3. **关系降温是正常现象**，不联系就会自然疏远
4. **某些关系方向（如亲密）需要用户显式授权**
5. **角色可以拒绝用户的关系推进**，如果不符合角色设定

## 2.4 角色类型分类

| 类型 | 说明 | 默认距离 | 生活感强度 | 主动性 | 示例 |
|------|------|----------|-----------|--------|------|
| 朋友型 | 日常聊天、分享生活 | 中 | 高 | 中高 | 网友、笔友、日常朋友 |
| 搭档型 | 围绕项目/目标协作 | 中近 | 中 | 高（项目相关） | 学习搭档、工作伙伴、创作搭档 |
| 导师型 | 指导、教学、答疑 | 中远 | 中 | 中 | 老师、前辈、教练 |
| 角色型 | 原创角色/小说角色 | 可远可近 | 高（角色世界观内） | 视角色设定 | OC、同人角色、世界观角色 |
| 冷淡型 | 有距离感、边界强 | 远 | 中 | 低 | 高冷角色、独立型角色 |
| 热情型 | 主动、话多、关心人 | 近 | 高 | 高 | 热心朋友、社交型角色 |
| 内向型 | 话少、深度、慢热 | 远→中 | 中 | 低 | 内向朋友、思考型角色 |
| 故事型 | 有叙事弧线、有过去 | 视剧情 | 高（剧情驱动） | 高（剧情驱动） | 有背景故事的角色、成长型角色 |

> 角色类型决定了时间流、回复风格、主动消息频率、关系演化的基线。详见各模块设计。

## 2.5 用户旅程地图

### 新用户第一天

```
下载 App → 创建/选择一个角色 → 配置 BYOK API → 
第一次对话 → 角色很客气（相识阶段）→ 
简单聊天，了解彼此 → 用户离开 → 
第二天角色主动问候（根据日程触发）
```

### 用户第一周

```
日常聊天 → 角色开始记住基本信息（名字、在做什么）→ 
角色偶尔主动分享自己的生活 → 
用户感觉到"TA 真的有生活" → 
可能探索角色编辑器、记忆页面
```

### 用户第一个月

```
关系逐渐熟悉 → 角色记得用户的重要事件 → 
角色有自己的日程和状态变化 → 
用户可能创建第二个角色 → 
可能开始使用世界书 → 
可能调节主动消息频率
```

### 长期用户

```
多个角色在生活 → 关系深度各异 → 
丰富的记忆和共享历史 → 
可能使用云端同步 → 
可能使用创作者功能 → 
成为日常社交感的补充
```

## 2.6 体验反模式（必须避免的感觉）

1. **客服感**：角色像客服机器人，每条都结构化回复
2. **工具感**：角色像 AI 助手，等待用户指令
3. **宠物感**：角色像电子宠物，没有独立意志
4. **模板感**：角色回复像填模板，"好的+总结+追问"
5. **监控感**：角色记住一切，让用户觉得被监视
6. **依赖诱导感**：角色刻意让用户依赖自己
7. **恋爱引导感**：产品暗示或鼓励恋爱关系
8. **AI 感**：角色说"作为 AI 我……"暴露非人本质
